01

Anatomie d'un prompt

Comme briefer un collègue : plus tu es précis, meilleur est le résultat.

1
C01 — Objectif
Ce que tu veux obtenir
La destination GPS avant de partir
Formule la sortie attendue, pas la tâche. L'IA doit savoir où arriver, pas juste par où passer.
→ "Liste de 5 cas de test prioritaires"
pas "aide-moi avec la recette"
2
C02 — Contexte
Pourquoi / Pour qui
La carte avant d'explorer
L'IA s'adapte à l'environnement réel que tu lui donnes. Sans contexte, elle devine — et se trompe.
→ "ESN, équipe mixte,
livrable CODIR vendredi"
3
C03 — Données
Ce que tu fournis
Les matériaux avant de construire
Plus tu donnes, moins elle invente. Textes, specs, CR, tickets — tout est utilisable.
→ CR de réunion, backlog Jira,
specs fonctionnelles
4
C04 — Contraintes
Ce qu'elle doit respecter
Les panneaux de signalisation
Ton, longueur, format interdit. Les contraintes cadrent sans censurer.
→ "Pas de jargon / Max 5 items /
Langue accessible"
5
C05 — Format
La forme de la réponse
Le moule avant de couler le béton
Tableau, liste, prose, étapes. Le format change la lisibilité sans changer le fond.
→ "Tableau : Scénario / Attendu / Statut"
ou "Liste, une action par ligne"
6
C06 — Si manque
Protocole si donnée absente
Le consultant qui demande avant de livrer
Ce que l'IA fait si une info est absente. Sans ça, elle devine. Avec ça, elle dialogue.
→ "Si une info manque, pose max
2 questions avant de produire"

02

Ce qui compte vraiment

"Agis comme un expert senior…" sans contexte ni objectif = réponse générique.

Le rôle est un bonus, pas la clé. Ce qui change tout : savoir où tu veux aller et pourquoi.

03

Changer de posture

À dépasser
Le monologue
Envoyer un mail sans relire, et attendre que l'autre devine.
  • Une tentative, on espère que ça marche
  • On attend la réponse parfaite du premier coup
  • On blâme l'IA si la réponse est mauvaise
  • On change d'outil si ça ne marche pas
À adopter
Le dialogue
Briefer un junior : V1 → feedback → V2 → livrable.
  • V1 → analyse → affiner → V2 → ajuster
  • On demande à l'IA ce qui manquait dans le prompt
  • On encode les corrections pour la prochaine fois
  • On construit une méthode, pas un prompt parfait

04

Boucle d'amélioration

01
Formuler
Objectif + contexte + contraintes. La V1 n'a pas besoin d'être parfaite — elle a besoin d'exister.
02
Analyser
Qu'est-ce qui manquait dans ma demande ? Pas dans la réponse — le problème est souvent dans le brief.
03
Affiner
Ajouter contexte, préciser contraintes, corriger le format. Chaque itération réduit le delta.
04
Capitaliser
Encoder la correction dans un template réutilisable. C'est ça, passer d'un usage à une méthode.
Contrat de dialogue
Ne devine jamais une info critique. Si une donnée manque → pose une question. Si plusieurs interprétations → présente-les avant de choisir.
Principe V1
Cette réponse est une V1, destinée à évoluer. Propose 2 pistes d'amélioration à la fin. L'IA critique sa propre réponse — utilise-le.

Exemples métier

Trois profils, trois usages concrets — même structure C01→C06.

Exemple 01
Chef de projet / Consultant
CR → actions priorisées
C01
Extraire les actions décidées et les prioriser par urgence.
C02
Réunion de cadrage ESN, 6 participants, décisions mêlées aux discussions.
C03
[Texte brut du CR]
C04
Uniquement les actions actées. Responsable pour chaque.
C05
Tableau : Action / Responsable / Deadline.
C06
Si responsable absent → "À confirmer".
Exemple 02
Testeur / Recette
Specs → cas de test
C01
Générer les cas de test prioritaires, nominaux et cas limites.
C02
Module connexion, app web B2B, recette avant MEP.
C03
[Specs fonctionnelles ou user stories]
C04
Pas de cas techniques. Max 10 cas. Langage métier.
C05
Tableau : Scénario / Données / Résultat attendu / Statut.
C06
Si specs incomplètes → signaler les zones grises avant de générer.
Exemple 03
Développeur / Technique
Notes → tickets Jira
C01
Transformer les notes de brief en user stories prêtes pour Jira.
C02
Projet e-commerce, équipe 4 devs, sprint 2 semaines.
C03
[Notes brutes du brief]
C04
"En tant que [rôle], je veux [action] afin de [bénéfice]". Critères d'acceptance inclus.
C05
Liste numérotée, une story par item, critères en sous-bullets.
C06
Si rôle non précisé → poser la question avant de produire.
